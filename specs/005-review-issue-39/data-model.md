# Data Model & App Scaffold Structure

This document outlines the standard declarative application structure that is generated by the `create-app` command and parsed by the `seed` command.

## File System Scaffold

When `oxibase create-app <name>` is executed, it generates the following layout:

```text
<name>/
├── data/
│   └── 001_init.sql          # Creates standard app tables and inserts initial seed data
├── templates/
│   ├── layout.html           # The base HTML layout
│   └── index.html            # The home page template
├── routes/
│   └── web.json              # Declarative route definitions (easier to read/write than SQL)
└── functions/
    └── hello.rhai            # Custom scripting logic
```

## System Tables Managed by `seed`

The `seed` command is responsible for translating the file system structure into database records. It will ensure the following system structures exist:

1.  **Schema**: `routes`
    -   **Table**: `routes.definitions`
    -   **Columns**: `method`, `path`, `template_name`, `context_query`
2.  **Schema**: `templates`
    -   **Table**: `templates.source`
    -   **Columns**: `name` (filename), `content` (file content)
3.  **Schema**: `functions` *(Implicit, depending on how custom functions are registered in the future, typically a system table or catalog)*

## Seeding Sequence (Transaction)

1.  **Init System**: `CREATE SCHEMA IF NOT EXISTS routes; CREATE SCHEMA IF NOT EXISTS templates; ...`
2.  **Clear State**: Delete existing routes and templates so the state accurately reflects the current directory structure.
3.  **Load Data**: Execute all `.sql` files in `data/` in sorted alphabetical order.
4.  **Load Templates**: Insert file contents into `templates.source`.
5.  **Load Routes**: Read `.json` files, parse them, and insert into `routes.definitions`.
6.  **Load Functions**: Insert scripts into the system functions catalog.
